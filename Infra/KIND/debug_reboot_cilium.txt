## DIAGNOSIS

### Layer 1: Pattern Analysis

Problematic nodes: worker (172.19.0.2), worker3 (172.19.0.3), control-plane (172.19.0.6)
Healthy nodes:    worker2 (172.19.0.4), worker4 (172.19.0.5)

On HEALTHY nodes (worker2, worker4):
  - kube-proxy: Running
  - Cilium: Running (1 restart after reboot)
  - iptables: 184 rules including KUBE-SERVICES and CILIUM chains
  - Cilium deleteQueue: ~1 entry
  - cilium_host interface: present with pod CIDR routes

On BROKEN nodes (worker, worker3, control-plane):
  - kube-proxy: CrashLoopBackOff / Error ("too many open files")
  - Cilium: Unknown (pre-reboot pod) or CrashLoopBackOff (config init container)
  - iptables: Only 9 rules, no KUBE-SERVICES chain
  - Cilium deleteQueue: 257 / 257 / 130 entries (overflowed)
  - cilium_host interface: absent (Cilium never started)

### Layer 2: Root Cause #1 - Inotify Limit Exhaustion

FINDING: fs.inotify.max_user_instances = 128 (host-level kernel parameter)

Measurement:
  - All kind container processes run as root (UID 0)
  - Root user's inotify limit is shared across ALL 5 Kind containers + host
  - Total inotify instances in use after reboot: ~107+
  - Per-container: control-plane=28, worker=14, worker2=23, worker3=18, worker4=24
  - Total ~107, limit=128, leaving only ~21 instances available

After reboot, Kind containers restart in race-condition order. Worker2 and worker4
started their kube-proxy processes first and grabbed enough inotify instances.
The remaining 3 nodes (worker, worker3, control-plane) tried to start kube-proxy
AFTER the limit was nearly exhausted.

kube-proxy requires ~9 inotify instances at startup to watch configuration files.
With < 9 instances available, it fails with:
  E0618 11:30:14.751204 1 run.go:72] "command failed" err="failed complete: too many open files"

NOTE: The file descriptor limit (nofile) was NOT the issue - both containers had
      max_open_files = 1,073,741,816 (virtually unlimited). The 2048 shown by
      `ulimit -n` in a shell is the shell's inherited default, not the container limit.

### Layer 3: Root Cause #2 - Cilium DeleteQueue Overflow

FINDING: When Cilium's deleteQueue overflows, the kubelet cannot clean up old pod sandboxes

When a Kubernetes node reboots, all running containers stop. On restart:
1. Kubelet restarts all pod management
2. For each pre-reboot pod, kubelet calls CNI DELETE to clean up network interfaces
3. If the Cilium agent is not running yet, the CNI plugin queues these deletions
   in /var/run/cilium/deleteQueue/
4. If the queue exceeds its limit, Cilium CNI ABORTS all new deletions

deleteQueue sizes found:
  - zt-lab-worker:        257 entries (overflowed)
  - zt-lab-worker3:       257 entries (overflowed)
  - zt-lab-control-plane: 130 entries (overflowed)
  - zt-lab-worker2:       1 entry (healthy)
  - zt-lab-worker4:       ~1 entry (healthy)

With an overflowed deleteQueue:
  - kubelet cannot terminate old pod sandboxes ("deletion queue has too many entries; aborting")
  - kubelet cannot create new pod sandboxes (blocked by cleanup failures)
  - All pods on the node remain "Unknown" (pre-reboot state, never updated)

### Layer 4: Deadlock Chain

The core deadlock after reboot on the 3 broken nodes:

  1. kube-proxy fails → no iptables DNAT rules for kubernetes ClusterIP (10.96.0.1)
     CAUSE: inotify limit exhaustion (root cause #1)

  2. Cilium config init container fails → can't reach k8s API at 10.96.0.1:443
     CAUSE: kubernetes ClusterIP unreachable (kube-proxy not running)

  3. Cilium agent never starts → no Cilium socket (/var/run/cilium/cilium.sock)
     CAUSE: config init container failure (cause #2)

  4. Kubelet cannot clean up old pods → deleteQueue fills up
     CAUSE: Cilium socket missing + existing queue overflow (root cause #2)

  5. DaemonSet cannot create new pods → pod creation times out
     CAUSE: Admission webhooks (Gatekeeper, Istio) have ClusterIP services
             unreachable from API server (no Cilium eBPF, no kube-proxy, no routes)

The 2 healthy nodes escaped this deadlock because they started kube-proxy before
the inotify limit was exhausted, bootstrapping their Cilium properly.

### Layer 5: Additional Blocking Factor - Admission Webhooks

The kube-apiserver runs on control-plane (172.19.0.6). When DaemonSet controller
tried to create new Cilium/kube-proxy pods:

  - API server calls admission webhooks before storing new pod in etcd
  - Webhooks requiring HTTPS connections to ClusterIPs:
    * gatekeeper-webhook-service.gatekeeper-system.svc:443 (ClusterIP 10.96.216.157)
      failurePolicy: Fail (check-ignore-label webhook, only for namespaces)
      failurePolicy: Ignore (mutation.gatekeeper.sh, validation.gatekeeper.sh)
    * istiod.istio-system.svc:443 (ClusterIP 10.96.62.28)
      istio-revision-tag-default webhooks: failurePolicy=Fail, applies to pods
      in namespaces without istio.io/rev or istio-injection labels
  - Without Cilium eBPF or kube-proxy, ClusterIPs are unreachable from control-plane
  - Without pod CIDR routes, DNAT to pod IPs also fails
  - Cumulative webhook timeouts exceeded DaemonSet controller request deadline
  - Result: "Timeout: request did not complete within requested timeout - context deadline exceeded"

Key observation: cilium-envoy DaemonSet pods (5/5) were already Running because
they were created BEFORE this deadlock state, so they were not affected.

## REPAIR SEQUENCE

### Fix 1: Increase inotify limits (breaks deadlock #1)

Used a privileged Docker container to write to host kernel parameters:
  docker run --rm --privileged --pid=host alpine sh -c "
    echo 8192 > /proc/sys/fs/inotify/max_user_instances
    echo 1048576 > /proc/sys/fs/inotify/max_user_watches
  "

(sudo was not available without interactive terminal; privileged docker achieves same effect)

Result: kube-proxy on worker (pds87) started Running within seconds
        kube-proxy on worker3 and control-plane: force-restarted via kubectl delete

### Fix 2: Clear Cilium deleteQueue (breaks deadlock #2)

  docker exec zt-lab-worker sh -c "rm -f /var/run/cilium/deleteQueue/*.delete"
  docker exec zt-lab-worker3 sh -c "rm -f /var/run/cilium/deleteQueue/*.delete"
  docker exec zt-lab-control-plane sh -c "rm -f /var/run/cilium/deleteQueue/*.delete"

This cleared 257/257/130 stale deletion entries from the pre-reboot pods.
The entries were safe to delete because those containers had already been killed
by the host reboot; there were no live network interfaces to clean up.

After clearing, the kubelet could process pod lifecycle operations.

### Fix 3: Bootstrap ClusterIP routing on broken nodes

Since kube-proxy on worker3/control-plane were deleted (waiting to be recreated)
and Cilium wasn't running yet, a temporary manual DNAT was added:

  # kubernetes service ClusterIP
  docker exec zt-lab-worker3 iptables -t nat -A OUTPUT -d 10.96.0.1/32 -p tcp \
    --dport 443 -j DNAT --to-destination 172.19.0.6:6443
  docker exec zt-lab-control-plane iptables -t nat -A OUTPUT -d 10.96.0.1/32 \
    -p tcp --dport 443 -j DNAT --to-destination 172.19.0.6:6443

This allowed the Cilium config init container to reach the k8s API server.

### Fix 4: Force-delete Unknown Cilium pods

  kubectl delete pod cilium-2nvz5 cilium-2s8v6 -n kube-system --force --grace-period=0

This removed the old pre-reboot Unknown pods so the DaemonSet could create fresh ones.

The control-plane had no routes to pod CIDRs (10.244.x.x). Without these routes,

## WHY 3 NODES BROKE BUT 2 SURVIVED

The pattern (worker2 and worker4 survived, worker/worker3/control-plane broke)
was determined by a RACE CONDITION during post-reboot startup.

After the host machine rebooted:
1. All 5 Kind containers restarted nearly simultaneously (~1 minute)
2. Inside each container, systemd started kubelet and containerd
3. Each kubelet immediately started launching pod containers, including kube-proxy
4. kube-proxy at startup creates several inotify file watches

The host kernel's fs.inotify.max_user_instances = 128 applies to ALL root-owned
processes across all containers. Since Kind containers run as root, all 5 nodes
share this limit.

Worker2 and worker4 happened to start their kube-proxy processes early enough
to grab available inotify instances. By the time worker, worker3, and control-plane
tried, the pool was nearly exhausted. Their kube-proxy processes failed with
EMFILE ("too many open files") when trying to create inotify instances.

This is non-deterministic: on subsequent reboots, DIFFERENT nodes could fail.
The only reliable fix is to pre-set inotify limits persistently (see Prevention below).

================================================================================
## ROOT CAUSES SUMMARY

1. PRIMARY: fs.inotify.max_user_instances=128 (default) insufficient for 5-node Kind cluster
   - A 5-node Kind cluster with many workloads requires 200+ inotify instances
   - Default 128 causes random kube-proxy failures across nodes after reboot
   - This triggered a cascade of failures in networking-dependent components

2. SECONDARY: Cilium deleteQueue overflow after reboot
   - When Cilium is not running and many pods need CNI DELETE, the queue fills
   - Queue overflow blocks all kubelet pod lifecycle operations on the node
   - Stale queue entries must be manually cleared to unblock recovery

3. TERTIARY: No persistent pod CIDR routing without Cilium
   - Control-plane and worker3 lost all pod CIDR routes when Cilium wasn't running
   - Required temporary manual static routes for admission webhook reachability
   - This was automatically resolved once Cilium started (eBPF restored routing)

================================================================================
## PREVENTION / PERMANENT FIXES

Add to /etc/sysctl.d/99-kind-kubernetes.conf on the HOST machine:

  fs.inotify.max_user_instances = 8192
  fs.inotify.max_user_watches = 1048576
  fs.inotify.max_queued_events = 32768

Apply with: sudo sysctl --system

This ensures the limits survive reboots and are set before Kind containers start.

Alternatively, add to ~/.bashrc or a systemd service to apply before starting Kind.
